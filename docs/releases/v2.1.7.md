# LaunchPad v2.1.7

## v2.1.7 ship: External-infrastructure preflight gate (BL-364)

**v2.1.7 ships the external-infrastructure preflight gate.** Pre-v2.1.7, a typo in `CLOUDFLARE_API_TOKEN`, a Pages project pointed at the wrong repo, an unticked `[TBD]` in PRD.md, or a missing `NETLIFY_SITE_ID` GitHub Secret would surface 30 minutes into a `/lp-build` autonomous run, after `/lp-inf` had already burnt the implementation cycle and the failure landed at the `/lp-ship` deploy wall. v2.1.7 moves all those checks to a fail-fast gate at `/lp-build` Step 0.6 + `/lp-ship` Step 0.6 + the standalone `/lp-preflight` command, plus a `--section <path>` flag so multi-section projects can scope spec-completeness checks to the section being built.

User-visible deltas over v2.1.6:

- **`/lp-preflight` slash command + 6 bundled provider profiles (BL-364, P1).** Drop `.launchpad/preflight.config.yaml` naming any subset of `{cloudflare-pages, vercel, netlify, cloudflare-dns, namecheap-dns, spec-completeness}` and `/lp-preflight` verifies provider account, API token, deploy project, GitHub Secrets, DNS resolution, and `[TBD]`-free PRD + `## [<version>]` CHANGELOG heading + `status: approved|reviewed|built` section specs. Per-item stale windows (default 30-365 days) re-prompt the user when a confirmation goes stale. The file's absence is opt-out: projects that haven't configured external prerequisites skip preflight silently.
- **`/lp-build` Step 0.55 + 0.6 wiring.** Target section name resolves BEFORE preflight (Step 0.55), and the resolved path is passed to `lp_preflight.py --section docs/tasks/sections/<name>.md`. The `section-specs-approved` probe scopes to that file only. Multi-section projects with one approved target plus other sections in `shaped` / `planned` state no longer false-fail the gate.
- **`/lp-ship` Step 0.6 wiring.** Same preflight gate, no `--section` arg (project-wide ship-time scan). Drift catch: if a user invokes `/lp-ship` directly without going through `/lp-build`, the gate fires again.
- **Provider project probes report EXISTENCE only.** Cloudflare Pages, Vercel, and Netlify probes verify the project exists under the credentialed account. The C1 confirmation box covers the GitHub-repo linkage attestation. Programmatic linkage verification via the API `source.config.repo_name` / `link.repo` / `build_settings.repo_url` blocks is tracked as BL-367 (v2.1.8).
- **DNS probes parse IP answers via `ipaddress.IPv4Address`.** Both the Cloudflare-specific and generic DNS probes split the answer chain: IPv4 answers are checked against parsed networks (`104.16.0.0/12`, `172.64.0.0/13`); hostnames are checked against suffix lists (`.cloudflare.com`, `.pages.dev`, configurable per-project). A CNAME hostname like `104.16.cdn.example.com.` can no longer false-positive the IP-prefix check.

## Highlights

- **`lp_preflight.py` (engine + 15 probes + CLI).** Single-module implementation of the four check categories: A (auto-detect-silent), B (API-verified-with-credentials), C1 (user-confirmed-then-probe), C2 (user-confirmed-trust-only). `ProbeClients` dependency-injection seam keeps all tests hermetic (no network, no shell). `assert_preflight_ok(repo_root, target_section=None)` is the single callsite for `/lp-build` + `/lp-ship` markdown. `run_preflight` returns a `PreflightReport` with separate `failures` / `needs_confirmation` lists so the refuse-message can distinguish "checks ran and failed" from "user hasn't ticked the box yet". CLI exits 0 / 1 (checks failed or awaiting confirmation) / 2 (config/profile load error) for clean wrapping from slash commands.
- **6 provider profiles (`preflight-profiles/*.yaml`).** Each declares its checks with `id`, `category`, `title`, `setup_hint`, `stale_window_days`, optional `probe` + `args`. Override semantics: `.launchpad/preflight.config.yaml`'s `overrides:` block keys check IDs (`cloudflare-pages.api-token-valid`) and merges `stale_window_days`. The `spec-completeness` profile covers autonomous-ack existence, PRD `[TBD]`-marker scan, CHANGELOG `## [<version>]` heading, section-specs ship-ready status, and warn-only uncommitted-changes flag.
- **`--section <path>` CLI flag + `target_section` engine parameter.** When provided, the engine clones the `section-specs-approved` check with `args.section_path` set to the resolved path (CheckDefinition is frozen, so we use `dataclasses.replace`). The probe inspects only that file. `/lp-ship` and standalone `/lp-preflight` do NOT pass the flag and retain the all-sections scan.
- **`ipaddress` module for DNS classification.** New `_parse_dig_answer(raw) -> (IPv4Address | None, hostname)` helper classifies one answer line. The Cloudflare probe checks IP membership in `ipaddress.IPv4Network("104.16.0.0/12")` + `IPv4Network("172.64.0.0/13")`; hostname matches go through `endswith(".cloudflare.com" | ".pages.dev")`. The generic dns-resolves-via-cname probe also splits: hostnames match `expected_suffix`, IPs match `expected_prefixes`. A hostname can no longer match an IP-prefix entry.
