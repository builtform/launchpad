# LaunchPad v2.1.6

## v2.1.6 ship — Tier 2 stack-aware refactors + autonomous-ack gate generalisation

**v2.1.6 lands the 8 Tier 2 stack-aware refactors deferred from v2.1.5 (BL-345 through BL-352) plus an architectural fix to the autonomous-ack gate (BL-356).** The Tier 2 work makes /lp-bootstrap output coherent for non-TS-monorepo stacks (Astro, Next.js single-app, Python, Ruby, Hugo, Go); pre-v2.1.6, those users hit a cascade of broken renders (TS-monorepo allowlist rejecting their root files, pnpm commands on Python projects, fictional Astro routes in APP_FLOW.md). BL-356 closes a direct-invocation bypass of the autonomous-ack gate that the v2.1.5 ack design only enforced on `/lp-build`.

User-visible deltas over v2.1.5:

- **Stack-aware `check-repo-structure.sh` allowlist (BL-347, P0).** Pre-v2.1.6 the rendered structure-check script hardcoded LaunchPad's monorepo shape (`apps/`, `packages/`, `pnpm-workspace.yaml`) and rejected every legitimate framework file on greenfield single-app projects — `astro.config.mjs`, `next.config.ts`, `public/`, `src/`, `manage.py`, `Gemfile` all flagged as "non-whitelisted". v2.1.6 ships per-stack ALLOWED_DIRS / ALLOWED_CONFIGS data in `_structure_allowlists.py` injected by a sentinel-based enricher at /lp-bootstrap render time. Greenfield projects no longer hit a P0 first-commit blocker.
- **`.gitignore` / `.gitleaks.toml` / `.greptile.json` are stack-aware (BL-350, P3).** TS-monorepo cruft (`.next/`, `.turbo/`, `pnpm-lock.yaml`) moved out of the universal kernel templates and into per-stack data. Python / Ruby / Hugo / Go users no longer see irrelevant entries in their rendered ignore files. `.greptile.json`'s ignorePatterns sentinels are stripped on every render (sentinels live inside a JSON string value).
- **`<stack>-noop: run: 'true'` placeholder hooks dropped (BL-351, P3).** The astro / generic / nextjs_standalone / ts_monorepo lefthook fragments no longer ship a cosmetic `run: 'true'` placeholder. The fragments are comment-only stubs ready for future real hooks; the multi-stack composition test for `nextjs_fastapi`'s real Python-gates content is preserved.
- **Minimal APP_FLOW.md placeholder hints + BackendInfo.static_capable (BL-348 + BL-349, both P1).** Every adapter's `describe_app_flow()` previously emitted concrete fake routes (`["/", "/about", "/blog"]` for Astro; `["/", "/signin", "/dashboard"]` for ts_monorepo) that misled users into thinking they were detected. v2.1.6 emits `entry_routes=["/"]` plus an explicit `[Placeholder — add real routes via /lp-shape-section]` marker naming the stack's routing convention. `BackendInfo` gains a required `static_capable: bool` field so BACKEND_STRUCTURE.md can render "static site, no backend" framing for Astro static / Hugo / Eleventy / Expo / generic stacks instead of describing fictional endpoints.
- **lefthook + ci.yml `run: pnpm <cmd>` lines rewrite for non-TS stacks (BL-346 + BL-352, both P1).** New `_package_managers.py` data declares per-stack-family lefthook hook bodies (ts / ts_python / python / ruby / hugo / go). Two new enrichers rewrite `run: pnpm test` → `run: pytest`, `run: pnpm typecheck` → `run: pyright .`, etc., for non-TS primary stacks. TS-family projects ship byte-identical to v2.1.5 (the no-op path). Setup-action steps in ci.yml remain TS-shape for non-TS projects (documented limitation; structural workflow rewrite deferred to v2.1.7).
- **Stack-detector recognises Astro, single-app Next.js, plain Python (BL-345, P2).** Pre-v2.1.6 the detector emitted `generic` for single-app Next.js (no monorepo signal), didn't recognise Astro at all, and produced broken pnpm-based bootstrap output for plain Python / FastAPI / Flask projects (all mapped to `generic`). v2.1.6 adds Astro / single-app-Next.js / `python_generic` mappings plus surfacing of eleventy / expo / supabase frameworks (v2.2 active enum candidates).
- **Autonomous-ack gate covers every direct-invocation autonomous-write command (BL-356, P1).** The v2.1.5 gate enforced `.launchpad/autonomous-ack.md` existence only on `/lp-build`, even though `/lp-build` is a meta-orchestrator that chains `/lp-inf`, `/lp-resolve-todo-parallel`, and `/lp-ship`. Calling any of those three directly bypassed the gate entirely. v2.1.6 lifts the gate into a shared helper (`assert_autonomous_ack`) and wires every direct-invocation command to call it. The refuse-message now embeds a copy-pasteable starter template beginning with `# Autonomous Execution Acknowledgment` — first-time users hitting the refuse don't have to read HOW_IT_WORKS.md to understand what the file is for.

## Highlights

- **BL-347 — stack-aware ALLOWED_DIRS / ALLOWED_CONFIGS (P0).** `plugin_stack_adapters/_structure_allowlists.py` declares per-stack entries for every stack in `STACK_ID_ACTIVE_ENUM`. The kernel `check-repo-structure.sh.j2` template gains `STACK_AWARE_{DIRS,CONFIGS}_{BEGIN,END}` sentinel comments inside the bash arrays. `lp_bootstrap/stack_structure_check.py` enricher reads `stacks:` from `.launchpad/config.yml`, looks up additions, and splices them between sentinels via regex substitution. `apps/` + `packages/` moved OUT of the kernel ALLOWED_DIRS and INTO the `ts_monorepo` stack entry — non-monorepo projects no longer inherit the monorepo shape.
- **BL-350 — stack-aware `.gitignore` / `.gitleaks.toml` / `.greptile.json` (P3).** `plugin_stack_adapters/_ignore_patterns.py` declares three per-stack dicts; `lp_bootstrap/stack_ignore_patterns.py` enricher splices entries between sentinels on each file. gitignore + gitleaks behave like BL-347 (greenfield → kernel bytes unchanged). greptile.json's sentinels live INSIDE a JSON string value, so the enricher ALWAYS runs and consumes the sentinels to keep the rendered ignorePatterns field valid JSON.
- **BL-351 — drop cosmetic placeholder hooks (P3).** astro / generic / nextjs_standalone / ts_monorepo fragments stripped of `<stack>-noop: run: 'true'` placeholders; replaced with comment-only header stubs ready for future real hooks. nextjs_fastapi fragment unchanged (real Python-gate content via shared partial). Three existing tests updated to assert absence (was asserting presence).
- **BL-348 + BL-349 — adapter contract additions (P1).** `BackendInfo` TypedDict gains a required `static_capable: bool` field with docstring explaining renderer semantics. All 10 adapter files updated (astro, eleventy, expo, fastapi, generic, go_cli, hugo, python_django, rails, ts_monorepo). Every adapter's `describe_app_flow()` (when non-None) emits the BL-348 placeholder pattern.
- **BL-346 + BL-352 — package-manager-aware lefthook + ci.yml (P1).** `_package_managers.py` declares `STACK_FAMILY` (ts/ts_python/python/ruby/hugo/go) + `STACK_LEFTHOOK_HOOKS` (per-family command bodies) + `STACK_CI_SETUP_ACTIONS` (per-family setup steps). Two enrichers in `lp_bootstrap/stack_pkg_manager.py` rewrite `run: pnpm <cmd>` body lines. Both no-op for TS family (kernel already correct) and for greenfield (no config → defaults to `ts`).
- **BL-345 — stack-detector extensions (P2).** `plugin-stack-detector.py` recognises `astro` dep → stack=`astro`; single-app Next.js (no workspaces, no turbo) → stack=`nextjs_standalone`; any Python project → stack=`python_generic` (Django still → `python_django`); `@11ty/eleventy` / `expo` / `@supabase/*` surfaced in the frameworks list as v2.2 candidates.
- **BL-356 — autonomous-ack gate generalisation (P1).** `plugin_stack_adapters/autonomous_guard.py` gains `assert_autonomous_ack(repo_root)` raising `AutonomousAckMissingError` if the file is absent + `assert_ack_not_same_commit_as(repo_root, section_name)` raising `AutonomousAckSameCommitError`. `AUTONOMOUS_ACK_TEMPLATE` constant carries the canonical starter template beginning with `# Autonomous Execution Acknowledgment`. Five command markdown files updated to call the shared helper: `/lp-build` 0.1+0.3, `/lp-plan` 0.3 + Step 5 transition, plus new Step 0.5 in `/lp-inf`, `/lp-resolve-todo-parallel`, `/lp-ship`. HOW_IT_WORKS.md gate documentation updated at 3 sites.

## Scope NOT in v2.1.6 (deferred to v2.1.7+)

- **BL-357 (P2): `/lp-shape-section --auto / --review / --batch` modes.** Stretch item documented in BACKLOG.md; held over to v2.1.7 with no v2.1.6 capacity-driven impact (the BLs above land cleanly without it).
- **Setup-action step rewrites in ci.yml for non-TS stacks.** BL-352 covers the `run:` body rewrites but leaves `pnpm/action-setup` + `actions/setup-node` setup steps as TS-shape for non-TS projects. Documented limitation in `lp_bootstrap/stack_pkg_manager.py` — automatic action swap would silently produce invalid workflows on any setup-action sha rotation. v2.1.7 candidate: structured workflow rewrite.
- **lefthook.yml prettier-fix / eslint-fix auto-fixers on non-TS stacks.** Pre-commit hooks operate on `{staged_files}` glob-matched JS/TS files. On a Python-primary project the globs match nothing — hooks are inert. Removing them requires structural template rewrite outside BL-346 scope.
- **astro.config.{js,mjs,ts} content-grep for output-mode detection.** BL-349 ships `static_capable=True` as the Astro default (matches Astro's own default of `output: 'static'`). Runtime detection of `output: 'server'` / `'hybrid'` from astro config is a v2.1.7 candidate.
- **Manifest tamper-detection composite SHAs for the 4 new enricher data modules.** The v2.1.5 BL-316 pattern extends `compute_source_template_shas` with a composite SHA only for `lefthook.yml` (kernel + per-stack fragments + shared partials). The 5 new enriched targets (`check-repo-structure.sh`, `.gitignore`, `.gitleaks.toml`, `.greptile.json`, `.github/workflows/ci.yml`) plus their data modules (`_structure_allowlists.py`, `_ignore_patterns.py`, `_package_managers.py`) ship without composite-SHA coverage in v2.1.6. A future v2.1.7 BL will extend the pattern with per-enricher `enumerate_*_template_dependencies()` helpers. The current state is no worse than v2.1.5 baseline — the kernel template SHAs still trip tamper-detection; only the data modules are uncovered.
- **End-to-end `run_bootstrap()` integration test for non-TS stacks.** Each new enricher is unit-tested with synthetic kernel bytes, but no test invokes the full `run_bootstrap()` engine pipeline against a project with `stacks: [python_django]` or similar persisted. The engine.py wiring is a 5-branch `if/elif` chain on `target_relpath` — straightforward to review, but a future refactor that moves enricher dispatch could regress without test coverage. v2.1.7 candidate.

The two architectural deferrals (manifest composite SHAs + e2e engine test) were surfaced by the /lp-review pass; both are real but parallel-scope. v2.1.6 ships the BL closure as designed; v2.1.7 covers the architectural hardening.

## Migration

No migration required. v2.1.6 is fully backward-compatible:

- TS-stack projects (the vast majority) ship byte-identical to v2.1.5 on rerun of /lp-bootstrap. Every new enricher no-ops for the `ts` family.
- Non-TS projects gain stack-appropriate output for the first time without any user action — the enrichers fire automatically when `.launchpad/config.yml` `stacks:` array names a non-TS stack.
- Existing `.launchpad/autonomous-ack.md` files continue to work; the gate widening just means `/lp-inf`, `/lp-resolve-todo-parallel`, and `/lp-ship` now check for the file on direct invocation. Users who already have the ack file see no change.

## Privacy convention preserved

Per the cross-session convention established during the v2.1.5 ship, v2.1.6 public-facing artifacts (PR description, commit messages, CHANGELOG, release notes) avoid referencing any specific consumer project or installation source by name. The motivation for each BL is described in generic terms — "real-world installed-plugin reproduction," "post-v2.1.5 architectural scope review," "stack-aware refactors deferred from v2.1.5" — without leaking private project details.

## Reviewer notes

- Codex / Greptile / `/lp-review` reviewers: every per-stack enricher has a fail-closed defense path. A malformed `.launchpad/config.yml` cannot break /lp-bootstrap — worst case is kernel-only output (which v2.1.5 was already shipping universally). The enrichers reduce to no-op on every error path; this is intentional.
- The single-source-of-truth invariant for the autonomous-ack gate is test-enforced: command markdown that contains the historical inline `IF .launchpad/autonomous-ack.md does NOT exist` pattern fails `test_gated_command_does_not_inline_absence_check`. Future contributors who copy-paste the v2.1.5 inline refuse-message will fail the lint.
- BL-345's "two implementations of stack detection" premise from the BACKLOG entry was speculative — the actual code only had one detector (plugin-stack-detector.py invoked via subprocess by lp_define_runner). The real BL-345 fix is detector coverage gaps, addressed in this release. A v2.2 BL will revisit the unified-module refactor if a duplicate detector surface emerges.
