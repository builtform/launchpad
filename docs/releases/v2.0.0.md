# LaunchPad v2.0.0

**Major release: greenfield project scaffolding pipeline.**

v2.0 introduces a four-command pipeline that takes a project description and produces a working scaffolded codebase with full architecture documentation: `/lp-brainstorm` → `/lp-pick-stack` → `/lp-scaffold-stack` → `/lp-define`. The pipeline is brownfield-aware (refuses to scaffold over existing projects) and chain-of-custody-bound (every step's output is SHA-256 envelope-signed and consumed by the next step). The brownfield path (`/lp-brainstorm` → `/lp-define` over an existing repo) continues to work unchanged from v1.x.

## Added

- **`/lp-brainstorm`** — describes the project, classifies the working directory as greenfield/brownfield/ambiguous, writes `.launchpad/brainstorm-summary.md`, and (for greenfield) drops a `.first-run-marker` sentinel that authorizes the empty-nonce-ledger first-run fast path.

- **`/lp-pick-stack`** — six-step decision engine: project classification → 5-question funnel (Q1 site-type / Q2 content-mode / Q3 backend? / Q4 language / Q5 deployment-target) → category match against an 18-entry catalog with 5 ambiguity clusters → manual-override resolver → rationale generation → signed `.launchpad/scaffold-decision.json` write. Embeds a structured `rationale_summary` array (sanitized, NFKC-normalized, defense-in-depth filtered) so `/lp-scaffold-stack` and `/lp-define` never read raw rationale.md (closes prompt-injection paths).

- **`/lp-scaffold-stack`** — consumer pipeline: 12-of-13 decision validation rules (rule 12 `brainstorm_session_id` deferred to v2.2 per BL-235) → atomic marker consumption → layer materialization across the 10-stack catalog → cross-cutting wiring (pnpm/turbo/lefthook for monorepos) → secret scan → atomic receipt write → nonce ledger append → telemetry. On failure, emits a structured `scaffold-failed-<ts>.json` with `recovery_commands` array (write-time destructive-path denylist applied; runtime enforcement deferred to v2.2 per BL-231).

- **6 new `/lp-define` adapters** — `astro`, `fastapi`, `rails`, `hugo`, `eleventy`, `expo`. Adds the v2.0 catalog stacks alongside the existing v1.x adapters (`ts_monorepo`, `python_django`, `go_cli`, `generic`, `polyglot`).

- **10-stack catalog** (`plugins/launchpad/scaffolders.yml`): `astro`, `next`, `eleventy`, `hugo`, `hono`, `fastapi`, `django`, `rails`, `supabase`, `expo`. 18-category routing catalog (`plugins/launchpad/scripts/lp_pick_stack/data/category-patterns.yml`) with 5 ambiguity clusters. Per-stack knowledge-anchor pattern docs under `plugins/launchpad/scaffolders/<stack>-pattern.md`.

## Migration

If you have an unconsumed `.launchpad/scaffold-decision.json` from a v2.0-rc install (with `version: "0.x-test"`), delete it and re-run `/lp-pick-stack` after upgrading. The pipeline rejects rc-version decisions at v2.0.0 with a structured remediation hint:

```
delete .launchpad/scaffold-decision.json and re-run /lp-pick-stack
```

The 4-hour replay window means a fresh re-run regenerates against the bumped writer constants without any hand-editing.

## Upgrading from v1.x

```
/plugin marketplace update builtform
/plugin uninstall launchpad --scope <same-scope>
/plugin install launchpad@builtform --scope <same-scope>
```

Restart Claude Code after updating. The brownfield `/lp-brainstorm` → `/lp-define` path works identically to v1.x; the new greenfield 4-command pipeline activates only when `/lp-pick-stack` runs against an empty (or `.git`-only) directory.
