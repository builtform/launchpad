# LaunchPad v2.1.4

## v2.1.4 ship — install-layout P0 unblock + bring-your-own-framework path + Astro sub-template walk-scope fix

**v2.1.4 unblocks the v2.1 greenfield pipeline for installed-plugin users and opens a clean bring-your-own-framework path.** Surfaced during a 2026-05-14 first-user greenfield dogfood test, which exposed three distinct ship-blockers (one P0 + two P1) for users running `/lp-pick-stack` → `/lp-scaffold-stack` against the Claude Code marketplace cache layout. Pre-v2.1.4 every manual-override scaffold-stack invocation against an installed plugin failed at catalog load — making v2.1.0/v2.1.1/v2.1.2/v2.1.3 effectively non-functional for real users on the install path.

User-visible deltas over v2.1.3:

- **`/lp-scaffold-stack` works for installed-plugin users (BL-327, P0).** The default catalog paths now resolve correctly under `~/.claude/plugins/cache/builtform/launchpad/<VERSION>/`. Pre-fix the engine looked for `~/.claude/plugins/cache/builtform/plugins/launchpad/scaffolders.yml` (extra `plugins/` segment, missing version subdir) and surfaced `catalog_load_failed` on every manual-override invocation. The `parents[2]`-based path arithmetic resolves the LaunchPad ROOT in BOTH source-repo AND install layouts.
- **AstroAdapter `blog` and `marketing` sub-templates scaffold cleanly (BL-328, P1).** The v2.1 D9.1 hardening's whole-tree symlink rejection no longer over-rejects pinned upstreams that ship test-fixture symlinks OUTSIDE the subtree the adapter actually copies. Cache fetch now accepts a `walk_scope` kwarg; AstroAdapter passes its `_SUB_PATHS` value so the disallowed-entry walk is scoped to `examples/blog` or `examples/portfolio` instead of the full withastro/astro tree (which contains test-fixture symlinks under `packages/`).
- **`generic` selectable as a primary stack via `/lp-pick-stack` manual-override (BL-331, P1).** Five new `(generic, role)` tuples (`frontend`, `frontend-main`, `frontend-dashboard`, `backend`, `fullstack`); `scaffolders.yml` widens 10 → 11 entries with a curate `generic:` entry; the manual-override menu surfaces `generic` as the 11th option. Use case: bring-your-own-framework — third-party Astro themes, custom starters, frameworks not yet supported by a stack-aware adapter (SvelteKit, Remix, Solid, Phoenix-LiveView). Distinct from the v2.2-candidate fallback path, which signals "stack-aware adapter not ready"; picking `generic` directly is positive intent.
- **Pin-rotation parity gate (BL-328 co-ship).** New `plugin-upstream-pin-walk-scope-parity.py` script clones every pinned upstream SHA at PR time and walks the actual subtree the adapter would copy; CI fails if any disallowed filesystem entry (symlink / block / char / fifo / socket) is found. Catches the BL-328 class of bug at PR time, not user runtime. Two pre-existing pins (vinta nextjs_fastapi, starlight astro/docs) waived in `KNOWN_BAD_PINS` with BL-329 + BL-330 tracking the rotations to v2.1.5 / v2.2.

## Highlights

- **BL-327 — `/lp-scaffold-stack` catalog path P0.** Re-rooted `DEFAULT_SCAFFOLDERS_YML` / `DEFAULT_CATEGORY_PATTERNS_YML` / `DEFAULT_PLUGINS_ROOT` at `parents[2]` of `lp_scaffold_stack/engine.py` (the LaunchPad ROOT in both source-repo and install layouts). Three regression tests at `tests/test_install_layout_catalog_paths_v214.py` pin the source-tree happy path, the install-layout path arithmetic, and the full E2E pipeline against a tempdir-built simulated install tree.
- **BL-328 — Astro symlink walk_scope.** Threaded an optional `walk_scope: str | None` kwarg through `template_cache.fetch()` / `verify()` / `_entry_files_match_manifest()`. Validated against traversal / absolute / NUL / oversize inputs at the cache boundary. AstroAdapter passes `walk_scope=_SUB_PATHS[sub_template_id]`. Security invariant ("no symlinks reach user project") preserved at the same boundary, just scoped to the actual copy path. 19 regression tests at `tests/test_astro_walk_scope_v214.py`.
- **BL-331 — generic-as-primary.** 5 `(generic, role)` tuples; `scaffolders.yml` `generic:` curate entry; new `plugins/launchpad/scaffolders/generic-pattern.md` knowledge anchor; `/lp-pick-stack.md` Step 4 menu update; v2-handshake-lint catalog-count assertion 10 → 11. 6 regression tests at `tests/test_pick_stack_generic_as_primary_v214.py`.
- **CI parity gate.** `.github/workflows/v2-handshake-lint.yml` gains a step running `plugin-upstream-pin-walk-scope-parity.py --offline-skip`. `docs/maintainers/upstream-pin-rotations.md` documents the gate.
- **5 BL entries appended (BL-327..BL-331).** All five surfaced from the same 2026-05-14 first-user greenfield dogfood test session.

## Migration from v2.1.3

No data migration required. v2.1.4 is purely additive: it unblocks workflows that were broken on the install path (BL-327), unblocks an additional sub-template path (BL-328), and opens a new bring-your-own-framework option (BL-331).

The two behavior surfaces to be aware of:

1. **Installed-plugin users were silently broken on `/lp-scaffold-stack` since v2.1.0.** v2.1.4 is the first version where manual-override scaffold-stack actually works against the Claude Code marketplace cache layout. Users on v2.1.3 or earlier who hit `catalog_load_failed` should upgrade to v2.1.4. No re-bootstrapping needed; the next `/lp-scaffold-stack` invocation will succeed.
2. **`generic` is now an explicit option in the manual-override menu.** Users who previously routed through the v2.2-candidate fallback to reach `generic` (passing `--accept-v22-fallback` against `python_generic` or similar) can now pick `generic` directly. The fallback path is preserved unchanged for users who want it.

### Known limitations

- **`nextjs_fastapi` and `astro/docs` still fail at runtime** (BL-329 + BL-330). The BL-328 walk_scope mechanism only helps adapters with a sub-template subtree. Both `nextjs_fastapi` and `astro/docs` consume the whole upstream tree, so the runtime D9.1 cache walk still rejects their pinned upstreams (vinta has `docs/CHANGELOG.md` + `docs/README.md` symlinks; starlight has root-level `README.md` symlink). Workaround until v2.1.5 / v2.2: pick `nextjs_standalone` (clean) and wire FastAPI manually, or pick `(generic, backend)` for the BYOF path.
- **The CI parity gate's `KNOWN_BAD_PINS` allowlist contains 2 entries.** v2.1.4 ships strict for FUTURE pin rotations but accepts the two pre-existing failures as known-bad. Future rotations adding to the allowlist must include an explicit BL ref.
