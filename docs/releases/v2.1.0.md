# LaunchPad v2.1.0

**Architecture refresh: composition wrapper, sealed identity, brownfield-safe re-runs.**

v2.1 layers a per-stack composition wrapper, a sealed identity contract, and a brownfield-aware re-run path on top of the v2.0 four-command greenfield pipeline. The user-visible surface adds two new commands (`/lp-bootstrap`, `/lp-update-identity`) and reshapes what happens when the pipeline runs a second time against an already-scaffolded project. Upgrading is in-place: v2.0 scaffold artifacts auto-migrate to `schema_version: "1.1"` on first contact with the v2.1 plugin, and the existing four-command greenfield pipeline (`/lp-brainstorm` → `/lp-pick-stack` → `/lp-scaffold-stack` → `/lp-define`) keeps its v2.0 behavior end-to-end.

## Highlights

- **Composition wrapper for multi-stack scaffolds.** A new wrapper at `plugin_stack_adapters/composition.py` resolves pair-table-from-data at runtime from each adapter's `composes_with` declaration, isolates per-stack dependencies, and handles workspace allocation with collision-suffix logic. The 5 active stack ids (`ts_monorepo`, `nextjs_standalone`, `nextjs_fastapi`, `astro`, `generic`) compose under an N=2 cap, with `nextjs_standalone + astro` and `nextjs_standalone + nextjs_fastapi` as canonical hot paths.

- **Sealed identity contract.** Identity values (`project_name`, `email`, `copyright_holder`, `repo_url`, `license`, `license_other_body`, `pii_opt_in`) are captured once at `/lp-pick-stack` time and sealed into `scaffold-decision.json` under a `schema_version: "1.1"` envelope. `/lp-update-identity` later edits sealed values atomically with byte-identical preservation of `generated_at` across re-seal.

- **Brownfield-safe re-runs.** The pipeline now detects re-entry against an already-scaffolded project and routes to one of five cases (no-op / seed-as-first-time / refresh-with-edits / Case D email cross-check / legacy `schema_version: "1.0"` migration), with a bidirectional sentinel cross-detect across `/lp-bootstrap`, `/lp-update-identity`, and `/lp-scaffold-stack` so two concurrent operations cannot corrupt each other's state.

- **`/lp-update-identity` for updating sealed identity values without re-scaffolding.** Edits any of the 7 identity fields, re-renders the 7 kernel templates (LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, README, SECURITY, AGENTS, CLAUDE) atomically, and prints a PII WARN noting that prior values persist in git history. See [docs/guides/IDENTITY_AND_PII.md](../guides/IDENTITY_AND_PII.md) for removal options.

- **`StackIdV22Candidate` forward-compat surface.** Five v2.2-candidate stack ids (`python_django`, `python_generic`, `nextjs_hono_cloudflare`, `nextjs_trpc_prisma`, `rails`) are now part of the closed enum at detect time, routing to `generic` with a verbatim INFO log. v2.2 will swap detection without renaming the enum.

## Migration from v2.0

When you install v2.1 against an existing v2.0 project, the plugin auto-detects the legacy `schema_version: "1.0"` envelope on first `/lp-bootstrap` or `/lp-update-identity` invocation and transparently migrates it to `schema_version: "1.1"`. The migration is performed in-memory first (so a failure halts before any disk write), then atomically re-sealed via `re_seal_decision_atomic` with `generated_at` preserved byte-identical.

### What auto-migrates

The following fields are added to the v1.1 envelope on migration. Existing v1.0 fields are preserved without change.

- `identity`: the 7-field identity block (seeded as `<placeholder>` values until `/lp-update-identity` runs)
- `identity_updated_at`: ISO 8601 UTC timestamp
- `kernel_render_state`: list of `{path, rendered_content_sha256, source_template_sha256}` entries (one per kernel template)
- `version_drift_log`: append-only list of `{from_version, to_version, fields_changed, applied_at}` entries

### Rollback policy

v2.1 does not support automatic downgrade to v2.0. The legacy v1.0 reader was retired in Phase 8.5; once a project's `scaffold-decision.json` carries `schema_version: "1.1"`, downgrading the plugin to v2.0 will produce a `version_unsupported` rejection. If you must roll back, restore `scaffold-decision.json` from git history before the v2.1 plugin's first run.

### Identity persistence in git history

`/lp-update-identity` updates the LATEST commit's identity but does NOT rewrite history. Prior identity values remain reachable via `git log -p` until you run `git filter-repo` over the relevant fields. See [docs/guides/IDENTITY_AND_PII.md](../guides/IDENTITY_AND_PII.md) for the canonical recipe.

### If you hit any of these, file an issue

- Re-entry detection misclassifies your project (e.g., a Case C no-op when you expected Case A refresh): include the output of `/lp-update-identity --dry-run`.
- `/lp-update-identity` refuses with `USER_EDIT_BLOCKS_REFRESH`: the engine detected a hand-edit to a kernel file. Use `--force` to override after reviewing the diff.
- Schema migration aborts before mutating disk: the legacy envelope shape did not parse cleanly. The migration is performed in-memory first, so a malformed envelope surfaces a structured precondition error (`SCAFFOLD_DECISION_MISSING` or `IDENTITY_VALIDATION_FAILED`) and the on-disk file is left untouched. Capture `scaffold-decision.json` and the structured error, file an issue, and restore from git.

## Added

Per-phase user-visible deliverables (Phases 2 through 10 of the v2.1 architecture migration):

- **Phase 2: `KernelRenderer` + 7 kernel templates.** A shared renderer base (`plugin_default_generators/_renderer_base.py`) provides Jinja2 env factories with autoescape posture pinned to defense-in-depth (HTML autoescape only on `.html`/`.htm`/`.xml`; StrictUndefined on every env), plus `shell_quote`, `to_yaml_safe`, and `markdown_safe` filters. The 7 stack-agnostic templates render LICENSE / CONTRIBUTING.md / CODE_OF_CONDUCT.md / README.md / SECURITY.md / AGENTS.md / CLAUDE.md with full canonical license bodies for MIT / Apache-2.0 / GPL-3.0 / BSD-3-Clause / ISC / MPL-2.0 (plus user-supplied body for `Other`).

- **Phase 3: `/lp-bootstrap` command.** Bootstraps the harness configuration (`.launchpad/agents.yml`, `.launchpad/config.yml`) from the plugin's bundled defaults, with a sentinel-protected execution path and read-running-plugin-version surface. Replaces the v2.0 template-clone flow with plugin-bundled overlay rendering.

- **Phase 4: Composition wrapper + Adapter Protocol.** Per-stack adapters now implement a `runtime_checkable` `Adapter` Protocol with `scaffold_into` + `apply_overlay` methods, an `unwrap_strategy` Literal for nested-Turborepo handling, and a `composes_with: dict[StackIdActive, CompositionRule]` declaration that drives the composition pair table at runtime. SHA-pinned upstream cache lifecycle, validation-before-flock ordering, and `MAX_CONCURRENT_FETCHES=3` semaphore round out the dispatch surface.

- **Phase 5: `/lp-build` dev stage + `safe_run_long_shell`.** New dev stage in the build orchestrator drives long-running subprocess invocations (CLI scaffolders, package installs) with SIGINT-safe process-group cleanup. The `--get-config-value` flag exposes harness config to shell scripts without re-running the full preflight.

- **Phase 6: Stack-aware review dispatch.** Review agents gain a `stack_scope` frontmatter field (16 `core_pipeline` agents, 13 `stack:any` agents, 6 `design_quality` agents, 1 `skill_quality` agent at v2.1). `/lp-review` and `/lp-harden-plan` filter agents per the detected stack(s) so a Python-only project does not load the TypeScript reviewer.

- **Phase 7: `StackIdV22Candidate` forward-compat enum.** Five v2.2-candidate ids land in the closed enum at detect time without changing routing behavior. Detection currently routes them to `generic`; v2.2 will swap detection per id without renaming the enum.

- **Phase 8: Decommission of `plugin-doc-generator.py`.** The 713-LOC v2.0 doc generator is removed in favor of the v2.1 `lp_define_runner.py` render-batch flow. Co-shipped CI gate prevents regression.

- **Phase 8.5: `lp_define_runner` render-batch + secret-scanner refuse-all.** New `render_batch` + `scan_batch` + `write_batch` flow gates every kernel and adapter render through a full-batch secret scanner; any finding refuses the entire batch atomically (no partial writes). `secret_allowlist.py` adds three suppression mechanisms (Jinja-comment, file-path-glob, regex). `BUNDLED_DEFAULT_PATTERNS` provides a fallback when `.launchpad/secret-patterns.txt` is absent.

- **Phase 10: `/lp-update-identity` command.** Edits sealed identity values atomically without re-scaffolding. Five-case re-entry detection (A through E) with transparent legacy v1.0 envelope migration handled before case dispatch. PII WARN print routed via the public `PII_WARN_LINES` constant. `KernelRenderer.refresh()` re-renders the 7 kernel templates with `prior_kernel_render_state` comparison so user-edited kernel files block refresh by default (override with `--force`).

## Security

LaunchPad's v2.1 defense-in-depth posture rests on six file-level invariants and one operational gate:

- **Sealed identity contract.** Identity values are captured once and sealed under `schema_version: "1.1"`. Edits to sealed values flow through `/lp-update-identity` with atomic re-seal; direct hand-edits to `scaffold-decision.json` are detected on next read and produce a structured `validate_identity` error.

- **Composition wrapper isolation.** Per-stack scaffold dependencies render into per-stack tempdirs under `composition_root/.tmp/lp-<stack_id>-<uuid>/` before atomic move into the final tree, so a hostile or buggy adapter cannot corrupt another adapter's workspace mid-render.

- **Persistent governance kernel.** Context, quality gates, and learnings carry across sessions through `.launchpad/` and `.harness/`, so each run starts informed instead of cold and the threat surface shrinks per session as the project's security posture matures.

- **Secret-scanner gate fires before disk writes.** The Phase 8.5 `render_batch` flow scans every rendered file before any `os.replace` runs. A single finding refuses the whole batch.

- **Audit log for v2.1 schema-source files.** CODEOWNERS extended with 8 v2.1 schema-source entries; modifications to schema constants require same-commit append-only audit-log entries verified by `plugin-v2-handshake-lint.py`.

- **CVE watch via `.github/workflows/cve-watch.yml`.** Nightly `osv-scanner` over the SHA-pinned upstream registry plus a tag-drift detector against `pin_registry.py` SHAs.

The PII WARN printed by `/lp-update-identity` reads:

```
WARN: prior identity values persist in git history (LICENSE, CONTRIBUTING.md, ...).
      See docs/guides/IDENTITY_AND_PII.md for removal options.
```

This is informational; the actual removal mechanism is `git filter-repo`. The `--quiet` flag suppresses the WARN print but does not change history-rewrite behavior.

CVE rotation policy is rooted in three shipped files:

- `.github/workflows/cve-watch.yml` (nightly scan + tag-drift detector)
- `docs/maintainers/RELEASE_PROCESS.md` (rotation checklist)
- `requirements.txt` (hash-pinned dependency manifest)

For tuning the secret scanner against false positives, see [docs/guides/SECRET_SCANNER_TUNING.md](../guides/SECRET_SCANNER_TUNING.md).

## Upgrading from v2.0

```
/plugin marketplace update builtform
/plugin uninstall launchpad --scope <same-scope>
/plugin install launchpad@builtform --scope <same-scope>
```

Restart Claude Code after updating. On the first invocation against an existing v2.0 project, `/lp-bootstrap` detects the legacy `schema_version: "1.0"` envelope and migrates it transparently to `1.1`. No manual intervention is required for the migration step itself.

### Plugin pinning recommendation

Pin `launchpad@v2.1.0` in your project's marketplace declaration so downstream installs do not auto-upgrade to v2.2 before you have validated the migration on your own working tree. Example block:

```json
{
  "plugins": [
    {
      "name": "launchpad",
      "version": "2.1.0",
      "source": "github:builtform/launchpad@v2.1.0"
    }
  ]
}
```

Once you have validated v2.1.0 against your project, drop the explicit pin (or bump to the next minor) on the next planned upgrade window. Pinning is the recommended posture for the migration window only; pinning forever defeats the auto-update path.
