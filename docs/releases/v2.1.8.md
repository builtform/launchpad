# LaunchPad v2.1.8

## v2.1.8 ship: autonomous-execution polish (BL-370 + BL-371 + BL-372)

**v2.1.8 closes three gaps that turned "autonomous /lp-build" into an interactive flow in practice.** Each gap surfaced from post-v2.1.7 scope-review reproduction on a real-world greenfield site build:

1. The v2.1.7 preflight gate (BL-364) was invisible to users who never authored `.launchpad/preflight.config.yaml` by hand. /lp-bootstrap did not propose it; /lp-build + /lp-ship silently skipped when the file was missing. The gate that was supposed to fail-fast on missing Cloudflare / DNS / secrets never fired for the majority of greenfield setups.
2. When preflight was configured, /lp-build Step 0.6 ran probes and then /lp-ship Step 0.6 ran the same probes again. Any C1/C2 confirmation that hit its stale window during /lp-inf + /lp-review + /lp-resolve-todo-parallel + /lp-test-browser would re-prompt the user at ship time.
3. Claude Code's permission model prompted the user for every Skill and Monitor invocation during the /lp-build chain. LaunchPad's autonomous-ack.md was a LaunchPad-internal gate; it did not register pre-approvals with Claude Code's permission system. Users hit 5-7+ permission prompts per /lp-build run.

User-visible deltas over v2.1.7:

- **`/lp-bootstrap` proposes a preflight config when deploy signals exist (BL-370, P1).** After the engine returns success, /lp-bootstrap scans the repo for `wrangler.{jsonc,toml,json}`, `vercel.json`, `.vercel/project.json`, `netlify.toml`, and `cloudflare/pages-action` / `cloudflare/wrangler-action` workflow references. When any signal is present AND `.launchpad/preflight.config.yaml` is absent AND the opt-out marker `.launchpad/preflight.config.skipped` is absent, the user gets a single y/N prompt to create the config with the detected providers plus `spec-completeness`. On accept, the config lands at `.launchpad/preflight.config.yaml` and subsequent `/lp-build` + `/lp-ship` invocations actually probe Cloudflare / DNS / secrets. On decline, a skipped-marker file is written so future bootstrap runs do not re-prompt.
- **Preflight receipt memoization between `/lp-build` and `/lp-ship` (BL-371, P1).** A successful `/lp-build` Step 0.6 pass writes `.launchpad/preflight-receipt.json` with the config + checklist SHA-256 and the freshness window (default 3600s; override via top-level `freshness_window_seconds:` in `.launchpad/preflight.config.yaml`). When `/lp-ship` Step 0.6 fires later, it reads the receipt and skips probes if the receipt is still valid (exit_code == 0, within freshness window, config + checklist SHA-256 unchanged). Stale, mutated-config, mutated-checklist, prior-failed, or corrupt receipts are invalidated automatically and probes re-run. The audit trail lands at `.launchpad/audit.log` with one line per receipt event (`preflight-skipped-via-receipt receipt_age_seconds=N writer=/lp-build` or `preflight-receipt-<reason>` for invalidations).
- **Claude Code permission-mode autonomy via `.claude/settings.json` merger (BL-372, P1).** When `.launchpad/autonomous-ack.md` exists, `/lp-bootstrap` proposes merging a bundled tool-level allowlist template (`Skill`, `Bash`, `Read`, `Write`, `Edit`, `Grep`, `Glob`, `Monitor`, `TodoWrite`, `WebFetch`, `WebSearch`) into the user's `.claude/settings.json`. The merger preserves existing customizations: tightened `Bash(git:*)` rules are NOT broadened to bare `Bash`; scalar values like `model` and existing `hooks` dicts pass through unchanged. After acceptance, the `/lp-build` -> `/lp-inf` -> `/lp-review` -> `/lp-resolve-todo-parallel` -> `/lp-test-browser` -> `/lp-ship` -> `/lp-learn` chain runs without per-Skill or per-Monitor permission prompts. Users who do not opt into autonomous mode (no `autonomous-ack.md`) see the existing prompts as before.

## Highlights

- **`lp_bootstrap.preflight_proposer` module.** Pure-function detector (`detect_deploy_providers`) + writer pair (`write_preflight_config` / `write_skipped_marker`) backed by `atomic_write_replace` (with `trusted_root` pin). Detection is scoped to the 6 provider profiles already shipped in `plugins/launchpad/preflight-profiles/` (`cloudflare-pages`, `vercel`, `netlify`, `cloudflare-dns`, `namecheap-dns`, `spec-completeness`); workflow-action hints that map to providers without a profile (e.g. github-pages) are intentionally absent so the generated config never names a profile that fails to load. CLI surface: `--json` for the slash command to read structured detection state, `--write-config --providers <comma-list>` for the accept path, `--write-skipped` for the decline path.
- **`lp_preflight.py` receipt artifact + read/write flags.** New module-level constants `RECEIPT_PATH`, `RECEIPT_VERSION = 1`, `DEFAULT_FRESHNESS_WINDOW_SECONDS = 3600`. New `check_receipt_validity(repo_root, freshness_window_seconds, now=None) -> ReceiptCheckResult` returns a structured reason (`valid` / `missing` / `corrupt` / `stale` / `config_changed` / `checklist_changed` / `prior_failed`) so the audit log can record exactly why a receipt was invalidated. `write_receipt` atomically writes on exit 0 and removes any stale receipt on nonzero so the next memoization-aware caller cannot silently bypass probes against a now-broken environment. The receipt is the ONLY atomic-replace surface in lp_preflight; the existing checklist writer remains a plain write (no concurrent-reader contract).
- **`lp_bootstrap.claude_settings_merger` module.** Tool-level template at `plugins/launchpad/templates/claude-settings-autonomous.json` (per-skill `Skill(launchpad:lp-inf)` syntax was assumed by the original brief but is not supported by Claude Code's current permission schema; this finding was verified via Context7 against `/anthropics/claude-code` docs). The merger's `plan_merge()` deep-merges in-memory and returns the list of dotted-path additions so the slash command can preview changes before the user confirms. The `_merge_permission_list` rule refuses to broaden a tightened user rule: if the user pinned `Bash(git:*)`, the merger does NOT add bare `Bash`. Scalar collisions resolve to the user's existing value (model, hooks dicts, etc. are preserved verbatim). `apply_merge` writes atomically via `atomic_write_replace` and refuses (exit 65) when `.launchpad/autonomous-ack.md` is absent so the autonomous-mode opt-in is never bypassed implicitly.
- **`/lp-bootstrap` post-bootstrap follow-up steps.** Two new sections in `plugins/launchpad/commands/lp-bootstrap.md` (Claude Code permission mode + preflight config) carry the decision matrix, prompt template, and the CLI invocations Claude reads from the `--json` summary output. The autonomy follow-up runs first so the autonomy + preflight prompts cluster together at the end of `/lp-bootstrap`.
- **`autonomous-ack.md` template extended.** `AUTONOMOUS_ACK_TEMPLATE` in `plugin_stack_adapters/autonomous_guard.py` gains a "Claude Code permission mode (BL-372)" section explaining the merger's existence, its preservation guarantees, and the prompt the user will see at next `/lp-bootstrap`. Existing 33 autonomous-ack tests still pass against the extended template.

## Architectural placement notes

The original BL-370 and BL-372 briefs suggested `plugins/launchpad/scripts/plugin_stack_adapters/deploy_target_detector.py` and `plugins/launchpad/scripts/plugin_stack_adapters/claude_settings_merger.py`. Both modules live in `lp_bootstrap/` instead because (1) the work is bootstrap-flow logic not stack-adapter logic, (2) the `atomic_write_replace` allowlist + the `/lp_bootstrap/` directory CODEOWNERS rule both already cover this directory, and (3) the existing allowlist entries (`policy.py`, `manifest_writer.py`) establish the architectural pattern that bootstrap-tier writers cluster under `lp_bootstrap/`.

BL-372's brief assumed `.claude/settings.json` "stays gitignored (already covered by recent privacy-audit additions)". It does not; the file is Claude Code's project-settings convention. Gitignoring it would lose the team-wide-sharing path. v2.1.8 leaves the gitignore state to the team's discretion; the merger operates on whatever state exists.

BL-372's brief also assumed Claude Code supports per-skill `Skill(launchpad:lp-inf)` allowlist entries. The Context7-fetched Claude Code docs only show TOOL-level entries (`Skill`, `Bash`, `Edit`, `WebFetch`, ...) plus the scoped `Bash(git:*)` form. The shipped template uses the tool-level shape; per-skill granularity is not currently supported by Claude Code's permission model. The merger refuses to generate per-skill entries even if requested.
