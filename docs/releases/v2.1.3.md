# LaunchPad v2.1.3

## v2.1.3 ship (this commit, post-PR-#66)

**v2.1.3 is a polish release.** No code paths change. One intentional behavior change lands as part of the `lp-commit` skill retirement (see migration section below) — it is a deliberate two-path workflow clarification, not a regression. The release exists to refresh the project's public documentation, correct skill metadata (a missing `user-invocable` frontmatter field on 16 process skills), retire one obsolete skill scaffold, and set up the v2.1.3 tag that Anthropic Marketplace points at for the initial directory submission.

The release is intentionally narrow. The originally-planned v2.1.3 hardening bundle (~22 items surfaced during v2.1.1 sweeps + Phase 4 deferrals + the corpus-trained reviewer agent cut from v2.1.2 scope) bumps to v2.1.4 so this lane can ship clean in a single review cycle.

## Highlights

- **README rewrite** (+259 lines). Replaced the terse one-line tagline with a structured pitch: a "The cold-session tax" framing of the problem and a 7-row competitive landscape table positioning LaunchPad against the current alternatives — Compound Engineering, Superpowers, BMAD-METHOD, Anthropic Code Review, GitHub SpecKit, AWS Kiro, Tessl, HumanLayer CRISPY, Agent OS, Continue.dev, Devin, OpenHands, Factory.ai, Ralph Loop, Claude Flow, and Goose. Each row names a real product, lists its genuine pros, and lists the cost it leaves on you, so a developer evaluating LaunchPad can see exactly where it fits in the agentic-coding landscape.

- **Skill metadata corrected.** Sixteen process skills (loaded by workflow commands, not user-invokable) gain a `user-invocable: false` frontmatter field: `lp-brainstorming`, `lp-compound-docs`, `lp-creating-agents`, `lp-creating-skills`, `lp-document-review`, `lp-frontend-design`, `lp-imgup`, `lp-prd`, `lp-rclone`, `lp-react-best-practices`, `lp-responsive-design`, `lp-step-zero`, `lp-stripe-best-practices`, `lp-tasks`, `lp-verification-before-completion`, `lp-web-design-guidelines`. Claude Code uses this field to decide which skills appear in user-facing skill lists. Net effect: cleaner skill discovery — users see only the skills they can directly trigger.

- **`lp-commit` skill retired.** The `/lp-commit` slash command remains unchanged. Only the redundant `SKILL.md` sidecar at `plugins/launchpad/skills/lp-commit/` (which duplicated the command's content) is gone, along with its empty `evals/` and `references/` scaffolds. **Behavior change for natural-language users**: the deleted SKILL.md previously routed phrases like "commit changes", "commit this", "commit my work", and "ready to commit" to `/lp-commit` automatically. Those natural-language triggers no longer fire — to commit, invoke `/lp-commit` explicitly. The slash command's quality gates, staging flow, and PR creation behavior are functionally identical. Anyone scripting against the old SKILL.md path should switch to the command file at `plugins/launchpad/commands/lp-commit.md`.

- **Methodology-attribution guidance.** `plugins/launchpad/commands/lp-create-agent.md`, `lp-create-skill.md`, `plugins/launchpad/skills/lp-creating-agents/SKILL.md`, and `plugins/launchpad/skills/lp-creating-skills/SKILL.md` gain a "Methodology attribution" section instructing agent/skill creators to use framework-citation form ("Based on [author]'s [framework]", "Operationalizes [author]'s methodology") rather than ingestion form ("faithful reading", "book-faithful", "ingested", "preserves exact terminology"). A verification grep recipe is included. Prevents attribution drift in future agent/skill generation.

- **`docs/growth/` folder allowlist.** A new top-level docs folder (`docs/growth/`) lands with a nested `.gitignore` that allowlist-tracks only `README.md` + `.gitignore` itself; everything else is gitignored. The tracked `README.md` is the public-facing pointer to the paid Growth Toolkit plugin; the rest stays out of git as internal strategy work.

- **BACKLOG/ROADMAP/CHANGELOG retargets** (Phase 0 of the release lane). v2.1.3 → v2.1.4 retargets across BACKLOG.md: 9 "defer to v2.1.3" decision lines, 7 "At v2.1.3 design time" headings, BL-297 heading (corpus reviewer doesn't fit polish scope), BL-323 in-body references. New BL-325 entry seeded for v2.1.4: `xargs -r` portability defense-in-depth (Codex flagged on PR #65 commit `0818c16`; empirically disproven on macOS BSD xargs but worth fixing in v2.1.4 as defense-in-depth for exotic Unix variants).

## Migration from v2.1.2

**Nothing to migrate.** v2.1.3 introduces no code paths and no schema changes.

The lefthook manifest source-template SHA (BL-316 Slice 4c.5 composite hash) does NOT change because the `_partials/_python_gates.j2.fragment` partial and the per-stack `lefthook.j2.fragment` files are untouched in v2.1.3. Existing consumer bootstrap-manifest.json files continue to verify clean — no MANIFEST_TAMPERED warnings, no `--accept-plugin-version-drift` needed.

Existing consumers see the SKILL frontmatter change only if they re-read the installed plugin files (e.g., when Claude Code re-indexes skill metadata). The `user-invocable: false` field is silently ignored by older Claude Code versions that don't recognize it.

The `lp-commit` skill removal has two distinct consumer surfaces:

**(1) Users scripting against the now-deleted `plugins/launchpad/skills/lp-commit/SKILL.md` file path.** Switch direct file references to `plugins/launchpad/commands/lp-commit.md`. The slash command file is unaffected; only the redundant SKILL.md sidecar is gone.

**(2) Users who relied on natural-language phrases** (`commit changes`, `commit this`, `commit my work`, `ready to commit`) **to trigger the gated `/lp-commit` workflow.** Those phrases no longer auto-route. This is **intentional two-path design**:

- To get the full LaunchPad commit workflow (branch guard + staging review + **mandatory dual-pass code review** + quality gates + structured commit message): invoke `/lp-commit` explicitly.
- To bypass the LaunchPad gates for a quick low-risk commit: continue using natural-language commit requests. Claude Code will fall back to its default commit handling (no LaunchPad review or gates fire).

The user picks per commit which path fits. If you exclusively use natural language and never type `/lp-commit`, you opt out of the v2.1.1 review hardening on every commit — that's the cost of the quick path, by design. The `/lp-commit` slash command itself is functionally unchanged.

## What stayed out

These items were considered for v2.1.3 scope and deferred:

- **BL-297 corpus-trained reviewer agent** (originally v2.1.2, then v2.1.3) → v2.1.4. Substantial code feature; doesn't belong in a polish release.
- **BL-313 hardening backlog** (~22 items from v2.1.1 sweeps + Phase 4 deferrals) → v2.1.4. Sentinel race conditions, identity-write hardening, additional semgrep invariants, doc-vs-code coherence patches, pyright strict mode on engine modules, false-positive cleanup, backup-retention command.
- **BL-324 layer-aware Python workspace probe** in `_python_gates` partial → v2.1.4. Theoretical regression at v2.1.x; v2.1.4 is the natural window when layer-override work lands.
- **BL-325 `xargs -r` portability** (defense-in-depth) → v2.1.4. Empirically disproven concern on the maintainer's target platforms (macOS + Linux + Alpine), but worth fixing for exotic Unix variants alongside the broader v2.1.4 hardening lane.

See [`ROADMAP.md`](../../ROADMAP.md) for the full v2.1.4 + v2.2 narrative.
