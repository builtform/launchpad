It covers subprocess invocation rules, governance (CODEOWNERS, CI lint, branch protection), branch drift mitigation, freshness policy, telemetry + Tier 1 reveal panels, cross-plan acceptance gates, and the amendment trail. The split exists so the contract layer stays small and stable; the operations layer absorbs ongoing changes (governance updates, telemetry adjustments, panel content tweaks) without requiring a contract bump. Both v2.0 implementation plans reference both documents; where they conflict, **the contracts doc wins**.

## 1. Subprocess invocation rules

Every subprocess call in v2.0 MUST go through the shared `safe_run()` helper at `plugins/launchpad/scripts/safe_run.py`. Internally the module is split into a pure-CPU argv-shape validator (`_validate_argv`) and a subprocess-invocation orchestrator (`_safe_run_invoke`); the public `safe_run()` calls both. The split lets unit tests cover argv allowlist rules without spawning subprocesses (the validator is a hot CI-lint surface; the invoker is the integration path).

1. **argv-list form only**: `subprocess.run([cmd, *args], shell=False, check=True, env=ENV_ALLOWLIST_VALUES, cwd=validated_cwd, capture_output=True)`.
2. **Validate `cmd` and each `args` element** against `^[A-Za-z0-9@._\-/=:]+$` via `_validate_argv()` (CI-lint enforced; unit-tested without subprocess fork).
3. **Strict env allowlist** (NOT a blocklist; unknown vars dropped):

```python
SAFE_ENV_ALLOWLIST = frozenset({
    "PATH", "HOME", "USER", "LANG", "LC_ALL", "TMPDIR",
    # Git terminal hardening: GIT_TERMINAL_PROMPT=0 is the only GIT_* allowed
    "GIT_TERMINAL_PROMPT",
})

# safe_run() OVERRIDES (not pass-through):
#   LC_CTYPE = "C.UTF-8"   # forced for byte-deterministic scaffolder output
#                          # across user shells; LC_CTYPE=C produces non-UTF-8
#                          # behavior in some scaffolders, which destabilizes
#                          # time_to_first_200 and secret_scan_passed metrics.
#                          # Layer 5 frontend-races P2-A9 honest scope note: this
#                          # override applies to spawned SUBPROCESSES only; the
#                          # orchestrator's own LC_CTYPE inherits parent. SHA-256
#                          # envelopes use ensure_ascii=True (locale-independent),
#                          # but user-facing error message reprs MAY vary by parent
#                          # LC_CTYPE. The "byte-deterministic" claim covers
#                          # subprocess output, not error message formatting.

# Documented forbidden vars (not exhaustive — anything not in allowlist is dropped):
# Code-injection vectors: NODE_OPTIONS, PYTHONSTARTUP, PYTHONPATH, PERL5OPT,
#   PERL5LIB, RUBYOPT, RUBYLIB, MIX_HOME, BUNDLE_*, GEM_*, CARGO_*, GOFLAGS, GOPROXY
# Loader hijacking: LD_PRELOAD, LD_LIBRARY_PATH, DYLD_*, LD_AUDIT
# Package-manager redirection: npm_config_*, NPM_CONFIG_*, YARN_*, PNPM_*
# Git hijacking: GIT_SSH_COMMAND, GIT_CONFIG_*, GIT_EXEC_PATH, GIT_DIR
# Plugin-internal: LP_*, *_DEBUG
```

4. **Set `cwd`** to a validated path via `path_validator.validate_relative_path()` (HANDSHAKE §6).
5. **Capture stdout/stderr; treat as untrusted-as-data.**
6. **Never use `shell=True`.** CI lint rejects any introduction.

The `SAFE_ENV_ALLOWLIST` constant is a frozenset module-level definition; CI lint asserts no other env construction occurs in subprocess call sites.

### Branch protection assertion

CODEOWNERS without enforced branch protection is decoration. **Three** gates verify continuously, not just at tag time (Layer 2 deploy F-03 + F-07 + Layer 5 adversarial P2-A8 + deployment P2-D3 hardening):

1. **PR-required-check on the squash-merge PR** (Layer 2 F-03 fix + Layer 5 adversarial P2-A8 extended to tag-protection): the v2-handshake-lint workflow runs the branch-protection AND tag-protection assertions as required checks on the squash-merge PR. If branch protection OR tag protection was disabled mid-PR (e.g., for a force-push fixup or admin "merge anyway"), the squash-merge cannot land. This closes the previous order-of-operations gap where the assertion ran AFTER merge AND the 24h watchdog gap where tag-protection drift was only caught daily. The `gh api repos/:owner/:repo/tags/protection` query uses the same fine-grained PAT (or split `TAG_PROTECTION_READ_TOKEN` per Layer 5 P1-S2) as the branch-protection check, with same fail-loud semantics: 401 → `tag_protection_token_unauthorized` (red); 200 with empty rules → `tag_protection_unreadable` (issue, no fail).

```bash
# Token scope (Layer 2 F-03 + Layer 3 security-lens P2-S2 + adversarial P1-RT-3):
# the default GITHUB_TOKEN cannot read repo administration endpoints. Use a
# fine-grained PAT stored as repo secret BRANCH_PROTECTION_READ_TOKEN, scoped to
# administration:read on this repo only, MAXIMUM 90-DAY LIFETIME, rotation logged
# in docs/releases/.
#
# Workflow MUST mask the token at runtime:
#   - run: echo "::add-mask::${{ secrets.BRANCH_PROTECTION_READ_TOKEN }}"
# AND restrict consumption to non-fork PRs only:
#   - if: github.event.pull_request.head.repo.full_name == github.repository
#
# The 401-vs-misconfigured distinction (Layer 3 security-lens P2-S2): emit
# `reason: "branch_protection_token_unauthorized"` for HTTP 401 (token expired
# or scope-downgraded — fail-LOUD: workflow goes red AND opens issue) vs
# `reason: "branch_protection_unreadable"` for HTTP 200 with null protection
# (config-issue — opens issue but does not fail-loud).
GH_TOKEN="$BRANCH_PROTECTION_READ_TOKEN" gh api repos/:owner/:repo/branches/main/protection \
  --jq '
    if (.required_pull_request_reviews // empty) | not then "ERROR: token cannot read protection"
    else
      .required_pull_request_reviews.require_code_owner_reviews and
      .required_pull_request_reviews.dismiss_stale_reviews and
      .enforce_admins.enabled
    end
  '
# Must return literal string "true". The empty-field guard catches token-scope
# failures (which would otherwise return null and silently boolean-coerce to false
# or — worse — pass through `&& true` chains as a silent green).
#
# PAT lifecycle runbook in docs/runbooks/branch-protection-token.md (Layer 3
# feasibility P1-5): rotation cadence, secret update procedure, fail-closed
# contract, recovery from expired-mid-PR scenario. Runbook captured as a
# Phase -1 deliverable.
```

2. **Phase 7.5 freshness pass** (pre-tag, gating): same `gh api` check, run again immediately before tag emission. Run **before** the version-bump commit lands (Layer 2 F-07 ordering: gates first → bump second → tag last); this prevents the failure mode where the bump commit lands in history with a misconfigured-protection environment.

3. **Nightly consolidated workflow** (`.github/workflows/v2-nightly-checks.yml`, Layer 5 performance P2-L5-1 + architecture P2-A3 compromise — consolidates 3 prior nightly cron triggers into 1 workflow with separate jobs to amortize runner spin-up while preserving cohesion): runs daily via `schedule:` with `concurrency: { group: v2-nightly-checks, cancel-in-progress: true }` to coalesce overlapping retries. Three independent jobs:
   - **`branch-protection`** job: same branch-protection check + content verification (rule pattern matches, `allow_deletions: false`, no force-push exceptions). Failure opens issue `security/branch-protection-misconfigured`.
   - **`tag-protection`** job: extends Layer 4 (deployment N2 + adversarial P2-RT4-G) — asserts tag-protection-rule existence + content via `gh api repos/:owner/:repo/tags/protection` (with fallback probe to `repos/:owner/:repo/rulesets`); asserts pattern matches broadened `v[0-9]+.[0-9]+.[0-9]+(-(yanked|recalled|rc[0-9]+|dryrun))?` regex (Layer 5 adversarial P1-A1) AND content rules per Layer 5 security P2-S3. Failure opens separate issue `security/tag-protection-misconfigured`.
   - **`branch-staleness`** job: replaces standalone `v2-branch-staleness-check.yml` — runs the `git log feat/v2-greenfield-pipeline..main --since=7.days.ago -- <handshake-touched paths>` query; opens issue `v2-merge-overdue` if non-empty.

Each job has independent failure semantics; one failing does NOT mask another. None block PRs (avoids self-DoS during reconfiguration). Catches drift during the 22-34-week dev window. The single workflow saves ~330-480 runner-minutes vs. the prior 3-workflow design while preserving per-concern cohesion. **Tag-protection rule creation + content verification is a Phase -1 manual step** documented in `docs/runbooks/branch-protection-token.md` (single runbook covers both branch + tag protection).

A unit test against a mock that returns 404 (token cannot read) MUST fail with a distinct message (`reason: "branch_protection_unreadable"`), not a generic boolean false. Test fixture lives in `plugins/launchpad/scripts/tests/test_branch_protection_assertion.py`.

### Forked-PR CI safety

**Two-layer defense** (Layer 2 F-05 + Layer 3 security-lens P1-S1 + adversarial P1-RT-3 hardening — pure maintainer protocol is unenforceable at single-maintainer scale):

1. **`v2-handshake-lint.yml`** is scoped to `pull_request` with **`permissions: { contents: read }`** (NOT `read-all` — Layer 3 security-lens P1-S1 minimum-privilege fix). Runs the full lint after maintainer-approves the workflow.
2. **`v2-handshake-lint-static.yml`** is scoped to `pull_request_target` with **`permissions: { contents: read }`** (NOT `read-all`; minimum-privilege). Runs ONLY the static lint against the merge-base diff — it does NOT check out the PR branch, so a malicious fork cannot exploit elevated permissions. Catches obvious garbage in fork PRs (shell metacharacters in `scaffolders.yml`, hyphen-prefixed test files, etc.) **before** maintainer-approves the full workflow run, breaking the "approval is the gate the lint is supposed to inform" circular dependency.

**Layer 3 explicit prohibitions for the static-lint workflow** (security-lens P1-S1 — closes the canonical `pull_request_target` CVE class):

- **Forbidden checkout pattern**: the workflow MUST NOT call `actions/checkout` with any `ref` derived from `github.event.pull_request.*` (head.sha, head.ref, merge_commit_sha, etc.). It checks out only the base ref or no ref at all (lints the diff via `gh api repos/:owner/:repo/pulls/<num>/files`).
- **Forbidden script behavior**: the lint script MUST treat fork-supplied diff content as data only. No `yaml.safe_load` or `importlib.import_module` on diff bytes; no `exec`/`eval` on fork-supplied content; no `subprocess` invocation with fork-supplied paths.
- **CI lint sub-rule** (Layer 5 schema-drift P2-SD5-4 reconciled — the AST `pull_request_target` shape check is INLINED into the default mode of `plugin-v2-handshake-lint.py`, NOT a separate `--check-pull-request-target-shape` flag; HANDSHAKE §12 dispatch contract is canonical): the default lint mode parses every `.github/workflows/*.yml` via YAML AST using `yaml.safe_load` (PyYAML pinned per Layer 5 security P2-S1), walks all `${{ … }}` expressions, and rejects any AST path containing `github.event.pull_request.{head.sha,head.ref,head.repo.*,merge_commit_sha,body,title,user.login}` or `github.event.workflow_run.{head_sha,head_branch}` — covers bracket-notation/`fromJSON(toJSON())`/expression-evaluator bypasses AND attacker-controlled fork-PR fields (Layer 5 security-auditor P2-4 extension). Also asserts `permissions:` is enumerated narrowly (no `read-all`/`write-all`); workflow-level default `permissions: {}` (zero) recommended with per-job opt-in.
- **`concurrency:` group**: pinned to `static-lint-${{ github.event.pull_request.number }}` with `cancel-in-progress: true` (Layer 5 frontend-races P3-L5-B + security-auditor P2-4 — closes the queue-stale-lint-comments race where force-push to fork PR would otherwise let the in-flight lint post a stale "PASS" comment after the new diff broke something).
- **Workflow-level `permissions: {}`** (Layer 5 security-auditor P2-4): set to empty (zero) at workflow root; per-job `permissions: { contents: read }` opt-in. Closes the future-amendment risk where adding a second job would inherit `read-all` from a workflow-level grant.

Behavior on forked PRs:

- **First-time contributors**: GitHub's default "Require approval for first-time contributors" causes `v2-handshake-lint.yml` to wait for maintainer approval. The static lint workflow has already run by this point; if it failed, the maintainer sees the failure and can decline to approve. Maintainer protocol: **never approve a forked PR's workflow run without locally checking out the branch and running `python plugins/launchpad/scripts/plugin-v2-handshake-lint.py` first**. A pre-commit hook helper `plugins/launchpad/scripts/check-fork-pr.sh` reduces this to one command (output is a single line `fork PR #X handshake-lint: PASS|FAIL`).
- **Returning external contributors**: full workflow runs automatically with read-only token. Cannot exfiltrate secrets or modify upstream files; lint output decides PR mergeability.

**Pre-tag audit** (Layer 2 F-05): before tagging v2.0.0, run `gh api repos/:owner/:repo/pulls?state=closed&base=feat/v2-greenfield-pipeline` and assert no merged-from-fork PRs touched any CODEOWNERS-gated path without a recorded "fork-lint-PASS" comment from the maintainer. The audit is best-effort defense in depth at single-maintainer scale; not equivalent to a security-team gating process — but acknowledged honestly rather than pretending the protocol alone suffices.

Document this two-layer defense in `CONTRIBUTING.md`.

### CI lint workflow

Single workflow file `.github/workflows/v2-handshake-lint.yml`, scoped to `pull_request` with `permissions: read-all`. Forked-PR CI runs WITHOUT write tokens to upstream, preventing exfiltration.

```yaml
on: [pull_request]
permissions: read-all
jobs:
  v2-handshake-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python plugins/launchpad/scripts/plugin-v2-handshake-lint.py
```

The lint validates:

- `scaffolders.yml` schema + every command/headless*flags element matches `^[A-Za-z0-9@.*\-/=:]+$`
- `category-patterns.yml` schema + every fits_when predicate is well-formed
- Every category's `canonical_stack[].stack` ∈ `scaffolders.yml` keys (cross-reference)
- Every scaffolder entry's `knowledge_anchor` file exists; SHA-256 matches `knowledge_anchor_sha256`
- All `last_validated:` dates present and within 90 days
- All scaffolder pattern docs are loaded only via `knowledge_anchor_loader.read_and_verify()` (grep)
- All subprocess calls go through `safe_run()` (grep for `subprocess.` not in helpers)
- No `shell=True` anywhere
- All test files under `plugins/launchpad/scripts/tests/` match `test_*.py` (CI-lint rejects hyphen-prefixed test files; CLI scripts must move to `plugins/launchpad/scripts/`)

### Phase 7.5 freshness pass (gating)

Pre-ship gate. CI runs `plugins/launchpad/scripts/plugin-freshness-check.py` (script is **promoted to a Phase -1 deliverable** per Layer 2 F-02 — it runs advisory on every PR for the entire 22-34-week dev window, so by Phase 7.5 ship time the script's behavior is well-exercised) and asserts:

- **Single 30-day freshness window** (Layer 3 simplicity P1-B + scope P2-1 — collapses Layer 2's 90d/30d/14d tier hierarchy into one window for v2.0): all `last_validated:` frontmatter on plugin-shipped catalog/pattern docs AND contract docs MUST be ≤ **30 days** from the tag commit. Release notes file existence is gated separately by the `feedback_release_notes_required` global rule (PR title `*vX.Y.Z*` requires `docs/releases/vX.Y.Z.md`); a freshness window for release notes is redundant with this existing gate. Tier-based freshness is deferred to v2.1 if observed drift problems demand it (BACKLOG **BL-213**).
- **Re-stamp affirmation pattern with chain-of-custody** (Layer 2 F-06 + Layer 3 data-migration P1-B + adversarial P1-RT-1 hardening): the only contract docs allowed to self-re-stamp during Phase 7.5 are `SCAFFOLD_HANDSHAKE.md` + `SCAFFOLD_OPERATIONS.md`, AND only via this enforced pattern:
  - The re-stamp MUST be in its own commit with subject matching `^chore\(v2\): re-stamp (HANDSHAKE|OPERATIONS|HANDSHAKE/OPERATIONS) last_validated, no contract change$` (Layer 3 deployment P3-A — accepts single-doc re-stamps cleanly).
  - The freshness-check script asserts: when `git diff --name-only HEAD~1 HEAD -- docs/architecture/` shows ONLY frontmatter `last_validated:` line changes (no body diff), the re-stamp commit MUST match the pattern AND carry a `Restamp-Affirmation: <prior-SHA> -> <new-SHA>` git trailer (parsed via `git interpret-trailers --parse` for canonical handling — Layer 3 pattern-finder P3-C).
  - **Chain-of-custody assertion** (Layer 3 data-migration P1-B): the freshness-check script parses `<prior-SHA>` from the trailer and asserts it points to a commit whose tree-hash for the re-stamped doc equals the parent commit's tree-hash for that doc. A re-stamp footer pointing at an unrelated SHA is rejected — closes the lineage-forgery vector that would otherwise let an attacker (or a force-push fixup) write an arbitrary affirmation footer that the regex match alone would accept.
  - **Schema_version handling at v2.0 — pinned (Layer 7 closes L6-λ #1 + L6-λ #2; Layer 8 simplified per code-simplicity P3; Layer 9 carve-out per schema-drift P3)**: v2.0 ships TWO distinct schema_version policies, scoped by file class:

    **Strict-policy files (3 v2.0 forensic surfaces — hard-reject on absent OR unknown)**: `restamp-history.jsonl`, `scaffold-rejection-<ts>.jsonl`, `tests/fixtures/manifest.yml`. v2.0-written lines carry `schema_version: "1.0"` from line 1; readers hard-reject absent OR unknown via single reasons `restamp_audit_schema_version_invalid` / `scaffold_rejection_schema_version_invalid` / `manifest_schema_version_invalid` (NOT silent-treat-as-v0; that v2.1+ reader policy is a forward-compat trap at v2.0). Single-reason design covers both failure modes because at single-maintainer scale both trigger identical recovery action ("delete the file and re-run"); v2.2 may reclaim the absent-vs-unknown distinction if telemetry shows confusion.

    **Lenient-policy files (2 v2.0 analytics surfaces — silent-skip-as-v0 acceptable)**: `.harness/observations/v2-pipeline-*.jsonl` (telemetry) + freshness report YAML frontmatter. Telemetry aggregator (`/lp-memory-report --v2-pipeline`) treats lines without `schema_version` as v0 with logged warning (matches the `--diff-from` cross-version-window semantics). Freshness report parser uses generic `freshness_drift_major` reason for verdict-level mismatch and is non-fatal at v2.0 (subsumed by the OPERATIONS §4 waiver mechanism for `verdicts:` entries). Telemetry+freshness are explicitly carved out from the strict-policy set because (a) they are non-load-bearing for security/forensic integrity (telemetry is opt-out via `telemetry: off`; freshness is advisory-with-waiver); (b) silent-skip preserves backward compat for the rare downstream user with stale `.harness/observations/` files from a v1.x project. v2.2 may unify both classes under strict-policy if telemetry shows the lenient policy was a footgun.

  - **Audit log** (Layer 3 data-migration P1-B + Layer 5 adversarial P1-A3 + security-lens P3-S2 + data-migration P3-DM5-3 baseline injection-defense at v2.0): every successful re-stamp commit additionally writes an entry to `.harness/observations/restamp-history.jsonl` via the `forensic_writer.write_restamp_audit()` call (NEVER via shell `echo >>` — closes commit-msg-hook injection vector where `\n` or `\r\n` in `$COMMIT_SUBJECT` would otherwise produce forged JSONL rows). The lefthook `commit-msg` hook MUST: (a) `json.dumps(payload, sort_keys=True, separators=(",",":"), ensure_ascii=True)` the entry — eliminates newline injection; (b) reject commit subjects containing `\n` or `\r` BEFORE JSON encoding with hook failure (better to block the commit than lose the affirmation entry); (c) write under `fcntl.flock(LOCK_EX)` on `.harness/observations/.restamp-audit.lock` (separate from telemetry lock to avoid DoS coupling); (d) file mode `0o600` via explicit `os.fchmod()`. Entry shape: `{schema_version: "1.0", commit, file, prior_sha, new_sha, author, timestamp, pid, pid_start_time}` — `schema_version` field ships at v2.0 (registered in HANDSHAKE §10 lifecycle bump list per Layer 5 P2-SD5-1) to avoid v2.0→v2.1 forward-compat trap; `pid`+`pid_start_time` provide minimal forensic identity (full chain-hashing deferred to BL-215). v2.1 readers MUST treat absent `schema_version` lines as v0 with logged warning. Provides structured query for "when was HANDSHAKE last semantically changed vs. just re-stamped?" months later.
  - Without this affirmation chain (subject + trailer + lineage match + audit log entry), the re-stamp is rejected — forcing the maintainer to either (a) write the affirmation explicitly with a real prior-SHA or (b) actually update the contracts.

- For `orchestrate` scaffolders: pinned command + version still resolves via the package manager (cached test outputs from throwaway-dir reruns).

**Telemetry payload byte limits** (Layer 3 frontend-races P3-B): each JSONL line MUST be ≤ 4096 bytes (PIPE_BUF on Linux); `materialized_files` arrays in failure entries are truncated to first 50 paths if longer. Enforces atomic single-`os.write` append safety.

**Lock + permissions** (Layer 3 security-lens P2-S7 + Layer 5 architecture P2-A2 split): `.harness/observations/.telemetry.lock` and `.harness/observations/.prune-progress` are created with mode `0o600` via explicit `os.fchmod()` after open (defends against world-readable umask). **`telemetry_writer.py` owns ONLY the analytics path** (`v2-pipeline-*.jsonl` + `.telemetry.lock` + `.prune-progress`); all forensic JSONL writes (security-events, scaffold-rejection, recovery-partial, restamp-history) are owned by the SEPARATE `forensic_writer.py` module per Layer 5 architecture P2-A2 SRP split — separate locks per concern (`.security-events.lock`, `.scaffold-rejection.lock`, `.recovery.lock`, `.restamp-audit.lock`) close DoS coupling where a long-held telemetry-prune lock would otherwise block forensic writes. Forensic paths are ALWAYS-WRITTEN (NOT gated by `telemetry: off`).

**Telemetry pruning ENOENT handling** (Layer 5 data-migration P3-DM5-1; Layer 8 ordering pin — closes frontend-races P3): if `.prune-progress` references a file that no longer exists (user manually `.gitignore`-cleaned `.harness/observations/v2-pipeline-*.jsonl` between two prune runs), skip with `os.stat` ENOENT as already-pruned, no error. **Lock-acquisition ordering**: the `os.stat(file)` for ENOENT detection MUST happen AFTER `LOCK_EX` acquisition on `.telemetry.lock`, NOT before — otherwise a concurrent writer creating a fresh `v2-pipeline-<ts>.jsonl` between the stat and the lock acquisition could be missed by this prune cycle (acceptable — caught next prune; but the spec must pin the ordering so implementers don't accidentally invert it for performance reasons). After completing the cycle, atomic-delete `.prune-progress` (rename to `.prune-progress.completed.<ts>`, retain max 5).

**Cross-lock acquisition prohibition (Layer 8 — closes frontend-races P3 lock-ordering; Layer 9 hardened — closes code-simplicity P3 wording + frontend-races P3 alias-bypass)**: v2.0 ships **3 file locks**: `.scaffold-nonces.lock`, `.restamp-audit.lock`, `.telemetry.lock`. (`.scaffold-rejection.lock` was eliminated by the Layer 8 simplification — see §8 amendment changelog row 8 for the audit trail.) v2.0 code paths each hold AT MOST ONE of these locks at any time — disjoint by design: nonce ledger ops in `/lp-scaffold-stack` consume path; lefthook commit-msg hook for restamp-audit; `telemetry_writer.py` for analytics tail. **Future cross-lock callers MUST follow a pinned acquisition order** (alphabetical by lock-file basename). v2.0 CI lint asserts `git grep -E 'fcntl\.flock|^\s*from\s+fcntl\s+import|^\s*import\s+fcntl\s+as' plugins/launchpad/scripts/` returns no call site outside the 3 sanctioned writers (`telemetry_writer.py`, `lp_pick_stack/__init__.py` nonce-ledger ops, lefthook commit-msg hook). The extended regex covers aliased imports (`from fcntl import flock`, `import fcntl as f`) that would silently bypass the literal `fcntl.flock` substring check. This forbids the next-layer regression of lock-coupling DoS.

### Qualitative interpretation rule (Layer 2 F-09 + F-12)

The four numeric thresholds (acceptance ≥60%, completion ≥80%, time-to-first-200 <90s, post-scaffold engagement ≥50%) are calibrated for ≥10 distinct project runs accumulated in `.harness/observations/`. At single-maintainer scale during the first 30-90 days post-ship, real-pipeline-runs may be 0-3, making numeric thresholds noise.

**Interpretation rule**: when N<10 distinct runs, the signals are **qualitative**, not numeric. The maintainer reviews each `.harness/observations/v2-pipeline-*.jsonl` entry individually against these questions:

- Did the pipeline complete without intervention?
- Did anything surprise me (manual edits to `scaffold-decision.json`, unexpected `/lp-define` output, dev server slow)?
- Was the recommendation acceptable for the project shape?

Each check-in (day 30, 60, 90 per BACKLOG entries) captures a `.harness/observations/v2-postship-day-XX.md` note. Numeric thresholds apply only when N≥10.

### Retention + atomic pruning (Layer 2 P2-2 + P2-3 + F-13)

Aggregator prunes JSONL entries older than 30 days at next pick-stack invocation. Bounds disk growth to ~30 days of activity. **Pruning protocol** (was unspecified; Layer 2 P2-2 + F-13 hardening):

- **Per-file atomic rewrite**: write surviving entries to `.harness/observations/<file>.tmp.<pid>` via `os.open(... O_WRONLY|O_CREAT|O_EXCL, 0o600)`; `os.fsync(tmpfd)`; `os.rename(tmp, file)`; `os.fsync(dirfd)`. Truncate-in-place is FORBIDDEN.
- **Lock** (Layer 2 P2-3): all telemetry writes AND prunes acquire `fcntl.flock(LOCK_EX)` on `.harness/observations/.telemetry.lock`. Lock is per-directory; prune holds the lock for the entire rewrite cycle. `telemetry_writer.py` (HANDSHAKE §12) is the single helper that owns this lock for both append and prune; concurrent commands race against the lock, not against each other's bytes.
- **Cross-file progress** (Layer 2 P2-2): list all `v2-pipeline-*.jsonl` files first, prune each in turn, log progress to a transient `.harness/observations/.prune-progress` file (written/`fsync`'d after each successful per-file rename). On next prune invocation, if `.prune-progress` exists, resume from the last-completed file. Partial application is acceptable (next run completes the prune); the only invariant is "no file is corrupted."

User can change retention window via `--retention-days <N>`.

## 6. Cross-plan acceptance gates

Both plans share these gates. Either plan failing a gate blocks v2.0.0 ship.

1. **Schema gate**: `scaffold-decision.json` round-trips through both plans without modification (pick-stack writes → orchestration reads → pick-stack re-validates from disk).
2. **Replay gate**: a `scaffold-decision.json` from one repo cannot be consumed in another (bound_cwd-triple mismatch). Test verifies hard rejection.
3. **Nonce gate**: the same `scaffold-decision.json` cannot be consumed twice (nonce already-seen). Test verifies hard rejection. Concurrent invocations do not race past the lock.
4. **Tamper gate**: mutating any byte of `scaffold-decision.json` after pick-stack signed it causes orchestration to hard-reject. Test verifies for: layers, paths, options, rationale_summary, generated_at, nonce, bound_cwd, sha256.
5. **Sanitization gate**: a malicious project description (containing `<script>`, fenced code blocks, URLs, `>>>` prompts, fullwidth-Unicode `＜＞`, RTL-override characters, zero-width joiners) reaches `/lp-define` only as filtered `rationale_summary` bullets — never as raw text in any architecture doc.
6. **Path-traversal gate**: a layer with `path: "../../etc/passwd"` is rejected at write-time (pick-stack) AND at read-time (orchestration). Defense in depth.
7. **Brownfield gate**: a brownfield cwd causes `/lp-brainstorm` to suggest `/lp-define`, AND `/lp-pick-stack` to refuse before asking questions, AND `/lp-scaffold-stack` to refuse if invoked anyway. Test verifies all three paths.
8. **Tier 1 reveal gate**: after a successful greenfield pipeline, the Tier 1 panel renders with all 5 enumerated governance items, each linking to the relevant file. Brownfield equivalent verifies the verified-mode panel.

Future automated recovery tooling (out of v2.0 scope; v2.1 candidate per BL-221) consumes `recovery_commands` (structured); humans read `recommended_recovery_action` (prose) and `see_recovery_doc` (URL). The `xargs rm` pattern from Layer 2 is REMOVED. Test asserts the structured fields are non-empty and reference real paths. **SIGKILL/SIGINT mid-materialization handling at v2.0 (Layer 8 — closes spec-flow P2-5)**: gate #11 covers `safe_run` failure on a layer (write `scaffold-failed-<ts>.json`, no auto-cleanup, nonce not consumed). User-initiated `SIGKILL` or `SIGINT` between layer-N file materialization and the receipt's `os.fsync` SKIPS the scaffold-failed JSON write entirely (signal-handlers cannot reliably write JSON during shutdown). Documented v2.0 behavior: the cwd is left with partial layer files, no rejection log, no failed-cleanup JSON, no consumed nonce, no marker rename. **User-discoverable via filesystem inspection only** — there is no audit-log breadcrumb. On re-run, the user must either (a) manually clean the cwd and re-invoke `/lp-scaffold-stack`, OR (b) `cd` to a fresh empty directory. v2.0 does NOT attempt SIGTERM/SIGINT graceful-shutdown handlers (would re-introduce surface area; explicitly out of scope).

**`/lp-define` brownfield receipt-missing warning text (Layer 8 — closes spec-flow P3-3; Layer 9 hardened — closes spec-flow P3 copied-marker edge case)**: the warning emitted when `scaffold-receipt.json` is absent MUST distinguish brownfield-expected from greenfield-corrupted. Brownfield happy path (no receipt because user ran `/lp-brainstorm` → `/lp-define` directly): warning text is "Brownfield mode: no scaffold-receipt.json found (expected — running /lp-define against existing project). Architecture docs will render from filesystem-detected stack." Greenfield-corrupted path (receipt expected per `cwd_state == "empty"` history but missing or sha256-invalid): warning text is "ERROR: greenfield receipt missing or corrupted; chain-of-custody lost. Re-run `/lp-scaffold-stack` to regenerate, or `cd` to a fresh directory." **Detection logic**: greenfield-corrupted requires BOTH (a) `.launchpad/.first-run-marker.consumed.<ts>` present AND (b) `.launchpad/scaffold-decision.json` present (consumed marker = scaffold-stack ran; decision file = pick-stack ran; both = chain-of-custody EXPECTED → receipt SHOULD exist). If consumed marker exists but decision file is absent, treat as orphaned audit trail from a copied `.launchpad/` directory (e.g., user `cp -r .launchpad/ <new-project>/` from a prior project) and fall through to brownfield-expected wording. The two-signal check closes the L9 copied-marker edge case while preserving the v2.0 detection simplicity.

**Scaffold-failed `reason:` closed enum (Layer 9 — closes adversarial P2)**: `scaffold-failed-<ts>.json` carries a SEPARATE closed `reason:` enum from `scaffold-rejection-<ts>.jsonl` (HANDSHAKE §4). v2.0 scaffold-failed reasons (closed set, enforced at write-time): `{layer_materialization_failed, auth_precondition_unmet, network_precondition_unmet, cross_cutting_wiring_collision, secret_scan_failed, recovery_precondition_unmet}`. v2.0 readers (humans + future BL-231 automated tooling) MUST hard-reject scaffold-failed entries with unknown `reason` values as `recovery_op_unsupported` (cross-references §4 enum's `recovery_*_unsupported` family). Distinct from §4 because scaffold-failed records POST-validation runtime failures (the validation already passed); §4 records PRE-execution validation rejections. Implementers reading both should NOT conflate the two enums — they live in different files and serve different forensic purposes.

## 7. Rollback procedure (Layer 2 F-11 + P1-4 → Layer 3 simplicity P1-E + scope P1-2 + product P2-1 compressed)

LaunchPad has no `prisma migrate down` analog. The rollback surface is git + marketplace pointer + downstream re-export. **Layer 3 compressed**: full operational runbook moved to `docs/maintainer/rollback-runbook.md` (maintainer-only); v2.0 contract spec retains only the load-bearing 4-step procedure + the pre-launch CI failure path. The full 6-step procedure + 24h observation window + decision tree + dry-run drill are in the runbook for the maintainer's stress-time reference; here we keep only what the contract spec needs to define.

### 7.0 Post-tag verification failure — `vX.Y.Z-recalled` rename (Layer 3 deployment P1-A + spec-flow P1-1 + Layer 4 spec-flow P1-3 + adversarial P1-RT4-B + security-lens P1-S3 — eliminated SHA-reuse exception)

If `verify-v2-ship` CI job (HANDSHAKE §12) fails post-tag, the recovery path **never reuses the version tag against a different SHA** (preserves HANDSHAKE §10 tag immutability invariant without exception). Layer 4 removed the prior 1-hour tag-deletion window because the propagation-status heuristic was unverifiable AND the SHA-reuse path violated immutability.

**Procedure** (applies regardless of time-since-tag — Layer 5 spec-flow P2-LF5 + adversarial P1-A1 + P2-A2 + security-auditor P2-3 reordered + hardened):

1. **Recall the failed tag**: `git tag v2.0.0-recalled <SHA-of-v2.0.0>` (NEW tag, original SHA — preserves history per HANDSHAKE §10). Push. **Pre-condition**: verify `gh api repos/:owner/:repo/git/refs/tags/v2.0.0-recalled` returns 404 (Layer 5 adversarial P1-A1 — closes namespace-squat where attacker pre-created the recall tag); if 200, hard-reject with `recall_tag_squat_attempt` security event + use next patch number's recall slot (`v2.0.0-recalled-r2`).
2. **Yank marketplace pointer FIRST** (Layer 5 spec-flow P2-LF5 ordering fix — flips to last-good BEFORE deleting GitHub release object so marketplace cache races toward "less broken" not "more broken"; same invert-ordering principle as §7.1 step 1): edit `.claude-plugin/marketplace.json`'s `latest` pointer back to v1.1.0; commit; push.
3. **Mark GitHub release as draft + delete assets**: edit GitHub release notes to add `~~v2.0.0~~` strikethrough + "recalled — verification failed" badge. Mark the GitHub release as a draft. Preserve assets to audit trail BEFORE delete: `gh release download v2.0.0 -D .harness/observations/recalled-release-v2.0.0/ || true`. Then delete assets via UI.
4. **Cancel orphan automation**: cancel any in-flight workflow runs scoped to the failed tag via per-id loop (NOT xargs — per-id error tolerance):
   ```bash
   gh run list --workflow=v2-release.yml --status=in_progress \
     --json databaseId --jq '.[].databaseId' | while read -r id; do
       # Self-exclusion (Layer 5 frontend-races P2-L5-A2): skip the run
       # executing the recall procedure if §7.0 was ever automated.
       [ "$id" = "$GITHUB_RUN_ID" ] && continue
       gh run cancel "$id" || true
     done
   ```
5. **Idempotent release-object delete**: `gh release view v2.0.0 >/dev/null 2>&1 && gh release delete v2.0.0 --yes` (Layer 5 security-auditor P2-3 — skip-if-absent; second invocation of recovery flow does not error on missing release).
6. **Revert bump-commit if landed**: revert the version-bump commit (marketplace.json + plugin.json + CHANGELOG.md as a SINGLE atomic revert per Layer 4 deployment N6 — never selective).
7. **Fix forward**: address the failed check on `feat/v2-greenfield-pipeline` (or fresh feat branch).
8. **Ship as next patch**: bump to `v2.0.1` (or `v2.1.0` if breaking change), re-run Phase 7.5 sub-steps from step 1 against the new version. **Never re-tag `v2.0.0`.**

**Local-clone remediation** (Layer 5 spec-flow P1-LF2 — closes downstream contributor stuck-on-stale-tag gap, since git tags are immutable client-side and `git pull` does NOT update existing local tags):

1. v2.0.1's release notes MUST contain a "If you have v2.0.0 locally fetched" subsection with the exact remediation command:
   ```bash
   git fetch --tags --force origin   # or your remote name
   ```
2. CHANGELOG.md's v2.0.0-recalled line carries the same hint.
3. The verify-v2-ship CI job's "no `0.x-test` residual" check is extended to also run `git tag -l v2.0.0` against the squash-merge SHA on the runner; if a stale `v2.0.0` tag exists locally on the runner that does NOT match the recalled SHA, fail with `reason: "stale_local_recalled_tag"` (defense in depth — if the maintainer's own clone has a stale tag, every release action they run is contaminated).

**What this trades**: one wasted version number per failed ship attempt. Benefit: `v2.0.0` always points at exactly one SHA in git history; downstream users who happened to install via cache between tag emission and recall get a consistent (broken) artifact, which they can roll back via OPERATIONS §7.1; they will never silently receive different `v2.0.0` content on a later install.

**User-facing recall communication checklist** (Layer 5 product-lens P2-PL5-3): (a) marketplace.json `latest` reverts to last-good version automatically (step 2 above); (b) GitHub release notes carry `~~vX.Y.Z~~` + recall reason; (c) `docs/releases/vX.Y.Z-recalled.md` is published with one-paragraph "what to do if you installed vX.Y.Z" guidance; (d) the next-shipped patch (vX.Y.Z+1) release notes link back to the recall AND include the local-clone remediation command above.

**Cache propagation observation** (Layer 4 deployment N3 — replaces the prior unreliable manual heuristic): the §7.0 path runs unconditionally on `verify-v2-ship` failure post-tag. If marketplace cache HAS propagated (some downstreams installed v2.0.0 between tag and recall), the recall + v2.0.1 ship is the same procedure — only the user-facing communication differs (release notes mention "if you happened to install v2.0.0 in the X-hour window: re-install v2.0.1 to receive the verified artifact").

### 7.0a verify-v2-ship propagation race at v2.0 (Layer 8 — closes adversarial P2-2)

v2.0 `verify-v2-ship` runs 4 checks single-shot per §0 strip-back: (1) tag SHA matches squash-merge HEAD; (2) plugin.json version matches tag; (3) no `0.x-test` residual; (4) no leakage regex. Check #1 hits the GitHub `gh api repos/:owner/:repo/git/refs/tags/<tag>` endpoint, which has the same ≤120s eventual-consistency window between squash-merge and tag-emission that BL-232 (exponential-backoff polling) is deferred to fix. **Documented v2.0 fallback**: if check #1 fails on first run with "tag SHA does not match HEAD," the maintainer waits 2 minutes and re-runs `verify-v2-ship` via `gh workflow run v2-release.yml --ref <tag>`. If three consecutive re-runs fail, escalate to §7.1 yank as a real failure. v2.2 BL-232 introduces in-workflow exponential-backoff polling; v2.0 ships manual retry as the operational discipline.

### 7.1 Compressed v2.0 rollback (4 essential steps)

For `verify-v2-ship` failures past 1h post-tag OR confirmed downstream-impact bug reports:

1. **Marketplace pointer FIRST** (Layer 3 adversarial P1-RT-1 — invert ordering vs Layer 2 spec; flips marketplace pointer back to v1.1.0 BEFORE yanking the tag, so cache-invalidation races toward "less broken" not "more broken"): edit `.claude-plugin/marketplace.json`'s `latest` pointer back to v1.1.0; commit; push.
2. **Yank tag**: `git tag v2.0.0-yanked <SHA-of-v2.0.0>` (NEW tag, not reuse — preserves history per HANDSHAKE §10 immutability). Push.
3. **Release annotation + CHANGELOG**: edit v2.0.0 GitHub release notes with `~~v2.0.0~~` + yank badge + 1-paragraph reason. Append to CHANGELOG.
4. **Open issue**: `v2.0.0 yanked: <reason>; remediation ETA <date>`. Document `LP_CONFIG_REVIEWED` rollback path: `unset LP_CONFIG_REVIEWED && claude /plugin install launchpad@v1.1.0 && export LP_CONFIG_REVIEWED=$(plugin-config-hash.py)`.

The full 6-step procedure (with 24h observation window, severity decision tree, what-rollback-does-NOT-undo, post-rollback v2.0.1 remediation, 3-patch-14-day escalation) lives at `docs/maintainer/rollback-runbook.md`. The runbook is referenced from this contract spec but maintained separately as operational doc, not user-facing API.

### 7.2 Un-yank procedure (Layer 3 deployment P2-B)

If v2.0.0-yanked turns out to have been a false alarm (e.g., the issue was a marketplace cache problem, not a code problem):

1. Edit GitHub release to remove strikethrough.
2. Restore `marketplace.json` `latest` pointer to v2.0.0.
3. Edit CHANGELOG to remove YANKED line.
4. Close the v2.0.0-yanked issue with explanation.
5. The `v2.0.0-yanked` git tag stays in history as evidence of the false-alarm episode (per HANDSHAKE §10 tag immutability).

### 7.3 24-hour post-tag observation window — monitored signals (Layer 5 deployment P1-D2)

The 24h observation window after a successful v2.0.0 tag emission is the maintainer's decision-making window for §7.0 vs §7.1 vs do-nothing. **Without explicit signal enumeration, the window is theatre.** The following signals MUST be monitored; the full procedural decision tree lives in `docs/maintainer/rollback-runbook.md`.

**Monitored signals**:

1. **GitHub Issues opened in 24h post-tag with label `severity/install-blocker`** (label MUST exist as Phase -1 setup; `gh api .../issues --label severity/install-blocker --since <T+0>` query in maintainer's check-in command).
2. **Manual install verification at T+1h, T+6h, T+24h**: `claude /plugin install launchpad@v2.0.0` in fresh `/tmp` scratch dir; capture full output to `.harness/observations/v2.0.0-postship-install-{1h,6h,24h}.log`. Any non-zero exit at any timestamp triggers escalation.
3. **`verify-v2-ship` re-run on a schedule** (every 4h for 24h via `workflow_dispatch` + `cron`) — guards against tag-protection drift between tag-time and final lock-in. Re-run failure triggers escalation per §7.0.
4. **CHANGELOG.md acknowledgment edit at T+24h**: maintainer flips `v2.0.0 (24h observation window open)` → `v2.0.0 (released)` if no escalation triggered. Flip to `v2.0.0 (recalled — see release notes)` if §7.0 fired during the window.
5. **GitHub Discussion or Issue requesting "how do I roll back?" within 24h** of the tag is treated as a §7.1 trigger candidate; if the report names a confirmed code-path bug, escalate to §7.0 instead.

**Decision matrix** (full version in runbook):

| Signal severity               | Window position | Action                                     |
| ----------------------------- | --------------- | ------------------------------------------ |
| Install-blocker confirmed     | <2h post-tag    | §7.0 recall (via `vX.Y.Z-recalled` rename) |
| Install-blocker confirmed     | 2h-24h post-tag | §7.0 recall + escalation issue             |
| Install-blocker confirmed     | >24h post-tag   | §7.1 yank + remediation patch              |
| `verify-v2-ship` re-run fails | any             | §7.0 recall                                |
| Cosmetic/UX issue             | any             | Patch in next minor (no recall)            |
